# vibe-writer 产品设计文档

**日期：** 2026-04-23  
**状态：** 待实现

---

## 一、产品定位

用户输入一个写作主题，vibe-writer 自动完成搜索、写作、审阅、配图，最终导出一篇完整的技术博客 Markdown 文件。

核心体验：**分阶段流程界面 + 关键节点用户介入**。

---

## 二、技术架构

### 整体结构

```
浏览器 (React)
    ↕ SSE / HTTP
FastAPI (Python)
    ↕
Agent Team (writer.py)
```

### API 端点（最小集）

| 端点 | 方法 | 说明 |
|------|------|------|
| `/jobs` | POST | 提交主题 + 介入节点配置，返回 `job_id` |
| `/jobs/{id}/stream` | GET | SSE 长连接，推送阶段事件 |
| `/jobs/{id}/reply` | POST | 用户在交互节点提交回复 |

### 中断处理

- **用户关闭窗口**：SSE 断开，agent 继续在后端运行。用户重新打开页面后重连，从当前进度继续（Checkpoint/Resume 模式，状态持久化到 `.tasks/`）
- **Agent 报错**：只重试失败的章节 subagent，不重跑整篇文章

## 三、Agent Team 设计

```
主 agent (Orchestrator)
    ├── 规划大纲
    └── 派发章节任务
            ↓
    ┌─────────────────────────┐
    │  章节 subagent（× N，并行）│
    │    ↓                    │
    │  搜索 agent              │  ← 判断资料质量，决定是否补搜
    │    ↓                    │
    │  写作（基于搜索结果）     │
    └─────────────────────────┘
            ↓
    审稿 agent（轻审：连贯性，每章完成后）
            ↓ 反馈给主 agent → 决定是否重写该章
            ↓
    所有章节完成
            ↓
    审稿 agent（重审：整体质量/可读性/正确性）
            ↓ 反馈给主 agent → 最终调整
            ↓
    导出 Markdown
```

### 角色职责

| Agent | 职责 | 类型 |
|-------|------|------|
| Orchestrator | 规划大纲、调度任务、处理审稿反馈 | 主 agent |
| 章节 subagent | 协调搜索和写作，产出单章内容 | subagent |
| 搜索 agent | 搜索资料、判断质量、决定是否补搜 | subagent |
| 审稿 agent | 轻审（连贯性）+ 重审（整体质量） | subagent |

> **设计原则：** 审稿 agent 的反馈对象是主 agent（Supervisor Pattern），而不是章节 subagent。"要不要重写"是调度层的决策。

## 四、前端界面设计

### 页面结构

```
┌────────────────────────────────────────┐
│  输入面板                               │
│  [主题输入框]  [介入节点选择]  [开始]   │
└────────────────────────────────────────┘
┌────────────────────────────────────────┐
│  流程面板                               │
│  Plan → 搜索 → 写作 → 审阅 → 配图 → 导出│
│  （当前阶段高亮，已完成打勾）            │
└────────────────────────────────────────┘
┌────────────────────────────────────────┐
│  交互面板（仅在用户选择的节点出现）      │
│  [Agent 输出内容]                       │
│  [用户对话输入框]  [确认继续]           │
└────────────────────────────────────────┘
```

### 介入节点配置

用户开始前选择一次，之后按选择执行：

- 大纲生成后（默认开启）
- 每章完成后
- 全自动（不介入）

### 交互节点行为

用户选择的节点到达时，流程面板暂停，交互面板出现：
- 展示 agent 当前输出（如大纲内容）
- 用户可以对话微调（如"把第三章改成讲实战案例"）
- 用户点击"确认继续"后，agent 继续执行

## 五、MVP 开发顺序

**第一阶段：跑通核心链路**
- FastAPI + React 基础框架
- 主 agent：输入主题 → 生成大纲 → 逐章写作 → 输出 Markdown
- SSE 进度推送
- 大纲节点用户确认交互

**第二阶段：接入搜索**
- 搜索 agent 集成 Tavily API
- 章节写作前先搜索

**第三阶段：接入审稿**
- 审稿 agent（轻审 + 重审）

**第四阶段：接入配图**
- DALL-E 配图 agent

**第五阶段：体验完善**
- 断点续写（Checkpoint/Resume）
- 章节重试机制
- 写作风格 Skill 支持
